# Plan: Multi-Repo Knowledge Graph with Service Topology

## Context

The Scality S3 platform is a microservice ecosystem with 11+ repos that interact via npm dependencies, HTTP APIs, Kafka events, and shared libraries. Three repos are currently indexed (cloudserver, Arsenal, backbeat) but the Neo4j graph is isolated per repo — you can't trace a request across service boundaries.

**Goal**: Build a multi-layer knowledge graph that enables queries like:
- "What would bucket notifications as a service need to integrate with?"
- "How does cloudserver authenticate requests via vault?"
- "What is the data flow for S3 PUT object from API to storage?"

## Graph Data Model

### Node Types

```
:Symbol    — Code-level: functions, classes, methods (existing)
             Properties: {file, name, repo, type, language}

:Service   — A deployable microservice
             Properties: {name, repo, port, protocol, description}
             Examples: "cloudserver", "backbeat", "vault", "bucketd"

:Endpoint  — An HTTP API route a service exposes
             Properties: {path, method, service, description}
             Examples: {path: "/_/crr/metrics/all", method: "GET", service: "backbeat"}

:Topic     — A Kafka topic
             Properties: {name, description}
             Examples: "backbeat-replication", "backbeat-metrics", "backbeat-replication-status"

:InfraService — Infrastructure dependency (not Scality code)
             Properties: {name, type}
             Examples: "redis", "kafka", "zookeeper", "clickhouse", "elasticsearch"
```

### Relationship Types

```
--- Service-level ---
:DEPENDS_ON       Service → Service           npm/library dependency (from package.json)
:HTTP_CALLS       Service → Service           HTTP communication (from config templates + code)
:USES_CLIENT      Service → Service           via client library (vaultclient, bucketclient, etc.)
:EXPOSES          Service → Endpoint          API routes the service defines
:PRODUCES         Service → Topic             Kafka producer
:CONSUMES         Service → Topic             Kafka consumer
:USES_INFRA       Service → InfraService      Infrastructure dependency (redis, zk, etc.)

--- Code-level (existing + new) ---
:RELATES          Symbol → Symbol             Intra-repo: CALLS, IMPORTS, INHERITS, CONTAINS
:XREF             Symbol → Symbol             Cross-repo code references (scoped by package.json)

--- Linking ---
:IMPLEMENTS       Symbol → Endpoint           Code function that handles an API route
:BELONGS_TO       Symbol → Service            Symbol is defined in this service
```

### Example Queries This Enables

```cypher
-- "What would bucket notifications need?"
MATCH (s:Service {name: "bucket-notifications"})-[r]->(target)
RETURN s, r, target

-- "How does cloudserver talk to vault?"
MATCH path = (cs:Service {name: "cloudserver"})-[*1..3]-(v:Service {name: "vault"})
RETURN path

-- "What handles PUT object end-to-end?"
MATCH (sym:Symbol {name: "objectPut"})-[:XREF|RELATES*1..4]-(related)
RETURN sym, related

-- "Which services would be affected if bucketd goes down?"
MATCH (s:Service)-[:HTTP_CALLS|USES_CLIENT|DEPENDS_ON*1..2]->(b:Service {name: "bucketd"})
RETURN s.name

-- "What Kafka topics does backbeat use?"
MATCH (bb:Service {name: "backbeat"})-[r:PRODUCES|CONSUMES]->(t:Topic)
RETURN t.name, type(r)
```

### Task 2: Create service analyzer module

**File**: `nullrealm/context/service_analyzer.py` (new)

Post-indexing analyzer with three sub-analyzers:

```python
@dataclass
class ServiceConnection:
    source: str           # "cloudserver"
    target: str           # "vault"
    rel_type: str         # "DEPENDS_ON", "HTTP_CALLS", "USES_CLIENT"
    properties: dict      # {package: "arsenal", port: 8500, via: "vaultclient"}

@dataclass  
class ServiceEndpoint:
    service: str          # "backbeat"
    path: str             # "/_/crr/metrics/all"
    method: str           # "GET"

@dataclass
class KafkaTopicUsage:
    service: str          # "backbeat"
    topic: str            # "backbeat-replication"
    role: str             # "producer" or "consumer"
```

Functions:
- `parse_package_json(repo_path)` → dep_map and ServiceConnections (DEPENDS_ON)
- `detect_client_patterns(chunks)` → ServiceConnections (USES_CLIENT)
- `detect_kafka_patterns(chunks)` → KafkaTopicUsage list
- `parse_federation_topology(federation_path)` → ServiceConnections + InfraService connections + port registry

### Task 5: Add MCP tools

**File**: `nullrealm/mcp_server.py`

New tools:
```python
@mcp.tool()
async def link_repos() -> str:
    """Create cross-repo symbol links and service topology edges.
    Run after indexing multiple repos to connect shared dependencies."""

@mcp.tool()
async def service_topology() -> str:
    """Show the full service topology: which services talk to which,
    via what protocols, what APIs they expose, and what infrastructure they use."""

@mcp.tool()
async def service_deps(service_name: str) -> str:
    """Show all dependencies of a specific service — upstream services it calls,
    downstream services that call it, Kafka topics, and infrastructure."""
```

### Task 6: Index remaining repos

Index in this order (via MCP `index_repo` tool):
1. Vault (closed, needs PAT with `auth_type="token"`)
2. MetaData (closed, needs PAT)
3. vaultclient, bucketclient, scubaclient, sproxydclient (open, small)
4. utapi, scuba (open, medium)

After all indexed, run `link_repos()` to create cross-repo XREF edges.

### Task 7: Index Federation (topology extraction only)

Federation is not code — it's Ansible playbooks. Instead of tree-sitter parsing:
1. Parse `group_vars/all` for service definitions (images, ports, defaults)
2. Parse `roles/run-*/templates/config.json.j2` for service connections
3. Generate Service, Endpoint, Topic, InfraService nodes directly
4. This runs as a special `index_federation()` function, not the regular `index_repo()` pipeline
