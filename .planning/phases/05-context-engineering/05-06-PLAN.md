# Plan: Index Federation as Config-as-Context (Phase 05-05 continuation)

## Context

Federation (development/10) is the deployment orchestrator for the entire Scality S3 platform. It contains 99 Jinja2 config templates, architecture documentation, tooling playbooks, and Python tools — critical context that can't be found in any code repo.

**Why this matters**: A question like "how do S3 analytics work end-to-end?" requires tracing: cloudserver access logs → log-courier → fluentbit → ClickHouse schema → aggregation MVs → Grafana. That chain is entirely in Federation config files, not in any code repo. Similarly, "how are bucket notifications configured?" needs `roles/run-bucket-notifications/templates/config.json.j2` — which shows the Kafka topic, ZK path, processor concurrency, and destination routing.

**Approach**: Treat Federation templates, docs, and configs as text chunks — embed in pgvector alongside code. Also extract service topology edges for Neo4j linking Federation configs to code repos.

## What to index from Federation

| Source | Count | Content | Chunk Strategy |
|--------|-------|---------|---------------|
| `roles/run-*/templates/*.j2` | 99 files | Service configs (JSON, nginx, SQL, YAML) | One chunk per file; split files >200 lines by sections |
| `group_vars/all` | ~1000 lines | Service definitions, ports, defaults, image map | Split by `defaults.X:` blocks — one chunk per service section |
| `documentation/*.md` | 4 files | Architecture docs (s3-analytics-pipeline, multicluster, etc.) | Split by `##` headers |
| `Components.md` | 1 file | Release/versioning model | One chunk |
| `tooling-playbooks/*.yml` | ~30 files | Operational procedures (account mgmt, backbeat control, metadata status) | One chunk per playbook |
| `tools/**/*.py` | ~15 files | Python tools (MetadataHealthCheck, MapReader, Leef) | Use existing Python parser |
| `env/maintainer/group_vars/all` | 1 file | Docker image registry mapping | One chunk |

## Federation → Code Repo Linking

### 1. CONFIGURED_BY edges
Each config template links to its Service node:
```
(:Service {name: "cloudserver"}) -[:CONFIGURED_BY]-> (:ConfigChunk {file: "roles/run-s3/templates/config.json.j2"})
```
The role name maps to a service: `run-s3` → cloudserver, `run-backbeat` → backbeat, `run-vault` → vault.

### 2. Service topology from config templates
Federation configs are the GROUND TRUTH for service connections. Parse `config.json.j2` templates to extract host/port references:
- `roles/run-s3/templates/config.json.j2` has `"bucketd": {"bootstrap": [...]}` → `cloudserver -[:HTTP_CALLS]-> bucketd`
- `roles/run-s3/templates/config.json.j2` has `"vaultd": {"host": ..., "port": 8500}` → `cloudserver -[:HTTP_CALLS]-> vault`
- `roles/run-backbeat/templates/config.json.j2` has `"zookeeper": {"connectionString": ...}` → `backbeat -[:USES_INFRA]-> zookeeper`

### 3. BUILT_FROM edges (Docker image → repo mapping)
From `group_vars/all`:
```
s3          → scality/s3, scality/cloudserver  (= cloudserver repo)
backbeat    → scality/backbeat               (= backbeat repo)
vault       → scality/vault, scality/vault-md (= Vault repo)
metadata    → scality/metadata               (= MetaData repo)
utapi       → scality/utapi                  (= utapi repo)
scuba       → scality/scuba                  (= scuba repo)
bucket_notifications → scality/backbeat      (= backbeat repo, same image)
```

Language detection for chunks:
```python
def _detect_config_language(filename: str) -> str:
    if filename.endswith('.sql.j2'): return "sql"
    if 'nginx' in filename: return "nginx"
    if filename.endswith('.yml.j2') or filename.endswith('.yaml.j2'): return "yaml"
    if filename.endswith('.conf.j2'): return "config"
    if filename.endswith('.json.j2'): return "json"
    return "jinja2"
```

### Task 2: Add `index_federation` MCP tool to `mcp_server.py`

```python
@mcp.tool()
async def index_federation(url="https://github.com/scality/Federation", 
                           branch="development/10",

    """Index Federation deployment configs into the knowledge graph.
    
    Indexes config templates, documentation, tooling playbooks, and group_vars.
```

Flow:
1. Clone Federation via `clone_or_pull(url, branch, auth_type="token")`
2. Call `index_federation(repo_path)` → chunks + service connections
3. Embed chunks via `embed_texts()` → store in pgvector with `repo="Federation"`
4. Store service connections in Neo4j via `store_service_graph()`
5. Register in repos table with `register_repo()`
